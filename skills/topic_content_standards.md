# Topic Content Writing Standards

Use this skill whenever writing or editing **educational content** for topic pages in this project. These rules govern voice, formatting, complexity, and structure.

---

## Core Principles

### 1. Write Like a Senior Dev Teaching a Friend
- Use first and second person ("you", "we", "your team")
- Be direct and conversational, not formal or academic
- Ask rhetorical questions and answer them in the next sentence
- Use specific real-world examples, not abstract concepts
- Sound like a smart person explaining something at a coffee chat, not a textbook

**Good:** "Here's the problem: your team will never agree on what to build next."
**Bad:** "Product teams face the fundamental challenge of reaching consensus on development priorities."

---

### 2. Never Use Em-Dashes or En-Dashes in User-Visible Text

**All em-dashes (`—`) and en-dashes (`–`) are banned** in strings rendered to the user. They render inconsistently across fonts, look odd at smaller sizes, and feel unnatural in reading flow.

#### Replace with natural alternatives:

| Instead of | Use |
|---|---|
| `X — Y` (adding context) | `X. Y` or `X, which Y` |
| `X — Y` (clarifying) | `X. In other words, Y` or `X: Y` |
| `X — Y — Z` (parenthetical) | `X (Y) Z` or rewrite as two sentences |
| `X — Y` (contrast) | `X. But Y` or `X, not Y` |
| `X — Y` (list intro) | `X: Y, Z, and W` |

**Examples:**
```
❌ "The split is random but consistent — the same user always sees the same version."
✅ "The split is random but consistent. The same user always sees the same version."

❌ "They handle the hard parts — user bucketing, analysis, SDK integrations — so you can focus."
✅ "They handle the hard parts like user bucketing, analysis, and SDK integrations so you can focus."

❌ "Failed experiments are not failures — they're answers."
✅ "Failed experiments are not failures. They're answers."
```

Note: Colons (`:`) and parentheses `()` are acceptable alternatives where natural.

---

### 3. Avoid AI-Sounding Language

These patterns make content feel like it was generated by AI. Avoid them:

| Avoid | Use instead |
|---|---|
| "It is worth noting that..." | Just say the thing |
| "This is particularly important because..." | Just explain why |
| "In the realm of..." | Just say "in" |
| "Gold standard for..." | Just describe what it does |
| "Leveraging X to Y" | "Using X to Y" |
| "Delve into..." | "Look at..." or "Understand..." |
| "Cutting-edge" / "robust" / "comprehensive" | Be specific instead |
| Passive voice everywhere | Use active voice |
| Listing 3 things with perfect parallelism every time | Mix it up naturally |

---

### 4. Content Complexity Levels

Every educational page should follow this structure:

1. **Main sections** — Accessible to anyone new to the topic. Use analogies, real-world stories, and plain language. Avoid jargon unless it is immediately explained.

2. **Optional Advanced section** — Collapsible, at the very bottom. Reserved for technical depth (statistics, implementation internals, advanced theory). Label it clearly with an "Optional · Advanced" badge and a subtitle like: "Not required, but worth the read when you're ready."

Never put complex theory in the main sections. If it requires prerequisite knowledge, it belongs in the Advanced section.

---

### 5. Use Real Examples Over Abstract Explanations

For every concept, find a real-world equivalent:
- Name actual companies (Booking.com, Netflix, LinkedIn)
- Reference real products and outcomes ("raised $60M more in donations")
- Use specific numbers ("4.1% follow rate, 5.9% follow rate, +44% lift")
- Reference plausible team scenarios ("Your designer wants a red button. Your PM wants a simplified checkout.")

---

### 6. Section Structure Template

Every section should follow this pattern:
```
Section Header (number + title + short subtitle)
  ↓
1–2 short paragraphs (the "what" and "why")
  ↓
Visual element (card grid, interactive diagram, accordion, etc.)
  ↓
Optional callout box (border-l-2 border-accent style) with a key insight
```

---

### 7. Interactive Diagrams — Story-Driven Content

When writing content for interactive step-through diagrams:
- Each step must tell a **story with a beginning, middle, and point**
- End each step with a "real lesson" callout that explains **why this matters in practice**
- Use specific numbers and outcomes in the story
- The user should feel like they're watching a real experiment unfold

**Template for each step:**
```
description: "What happened in this phase, with specific details and numbers."
realWorldNote: "The real lesson: what this means for you when you run your own experiment."
```

---

### 8. Formatting Rules for JSX/TSX Content

- **No orphan dashes** in string literals or JSX text nodes
- **Quotes in JSX text:** Use `&apos;` for single quotes and `&quot;` for double quotes
- **Bold highlights** (`<strong className="text-platinum">`) should be used for 2–4 words maximum per paragraph, not whole sentences
- Subtitle strings on `<SectionHeader>` should be ≤ 12 words and not contain dashes
- Tag meta strings (read time, track name) should not contain dashes

---

### 9. Tools and Platforms Section

When comparing tools:
- Start with the simplest/most beginner-friendly option
- Include an emoji per tool for visual scanning
- Tagline should be one sentence: "Best place to start if you're new to X."
- Description should tell the reader what type of team it fits, not just list features
- Avoid superlatives ("best", "most powerful") — use specific claims instead

---

### 10. Takeaways / Key Points

For takeaway card grids:
- 6 cards max, 3 per row at large breakpoint
- Each title should be a plain statement, not a category name
  - ✅ "Your gut feeling is not data"
  - ❌ "Data-Driven Decision Making"
- Each description should start with why it matters before saying what to do
